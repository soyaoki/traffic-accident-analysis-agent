## はじめに

「データ分析エージェント」という言葉を最近よく耳にするようになった。単にデータをSQLで引いてグラフを描くだけでなく、**自律的に仮説を立て、分析を実行し、次のアクションまで提案する**──そんなAIエージェントが実用段階に入りつつある。

本記事では、2026年現在のデータ分析エージェントの技術動向を整理した上で、実際に試した内容をお伝えします。

## Part 1: 動向調査

### 1-1. AIエージェント市場の現在地

AIエージェント市場は今、急速な拡大局面にある。市場調査によれば、Agentic AI市場は2025年時点で約73億ドル規模であり、2034年には約1,391億ドルへと年率40%超で成長すると予測されている。

ただし、現場の実態はまだ黎明期だ。2026年3月に650社を対象に行われた調査では、**78%の企業がAIエージェントのパイロットを持ちながら、本番スケールに到達したのはわずか14%** にとどまる。「テストでは成功するのに本番移行できない」という課題が世界中の企業で同時発生している。

本番運用を阻む主な壁は3つだ。

- **データ品質の問題**：AIの精度はデータ基盤の質に依存する。サイロ化されたデータや、コンテキストのない生データではエージェントは機能しない
- **ガバナンスの不整備**：誰がエージェントの行動を監視するか、失敗時の責任の所在が曖昧なまま本番投入するケースが多い
- **ROI証明の難しさ**：経営陣は「AIを入れた」ではなく、コスト削減・処理速度・精度向上の数値を求めている

### 1-2. データ分析エージェントの技術的な基盤

データ分析エージェントを理解するには、その背後にある基礎論文を押さえておく必要がある。

#### Chain-of-Thought (CoT) Prompting

2022年、Google BrainのWei et al.による論文 **"Chain-of-Thought Prompting Elicits Reasoning in Large Language Models"**[^1] が、LLMにおける推論能力の研究に大きな転換をもたらした。LLMに「中間的な推論ステップ（思考の連鎖）」を生成させることで、複雑なタスクの精度が大幅に向上することを示した論文で、CoT推論はモデルスケールに伴って**創発的（emergent）に発現する**特性であることも明らかにしている。

#### ReAct: Reasoning + Acting

CoTを実際の「行動」と組み合わせたのが、2023年にICLRで発表された **"ReAct: Synergizing Reasoning and Acting in Language Models"**[^2]（Yao et al., Princeton / Google）だ。LLMが**推論トレース（Thought）と具体的な行動（Action）を交互に生成する**フレームワークで、外部ツールやAPIと連携しながら多ステップのタスクを自律的に実行できる。このReActループが現在の多くのデータ分析エージェントの中核設計となっている。

```
Thought: 売上データをDBから取得して四半期比を確認する必要がある
Action: sql_query("SELECT quarter, SUM(revenue) FROM sales GROUP BY quarter")
Observation: Q1=1200, Q2=980, Q3=1350, Q4=1100
Thought: Q2が前後の四半期より落ちている。原因を調査する
Action: search_logs("Q2 anomaly revenue drop")
...
```

#### LLMエージェントのアーキテクチャ全体像

Wang et al. (2024) の **"A Survey on Large Language Model based Autonomous Agents"**[^3] はLLMエージェントの構成要素を4つに整理している。

| コンポーネント | 役割 | データ分析エージェントでの例 |
|---|---|---|
| **Profile** | エージェントの役割定義 | 「データサイエンティスト」としてのシステムプロンプト |
| **Memory** | 短期・長期の情報保持 | 会話履歴・ナレッジベース・分析結果のキャッシュ |
| **Planning** | タスク分解と実行計画 | 「まずデータ確認 → 異常検知 → 原因推定」の計画立案 |
| **Action** | 外部ツールとの連携 | SQL実行・Python実行・グラフ生成・Slack通知 |

また、データ分析領域に特化したサーベイとして Rahman et al. (2025)[^4] も参照されたい。

### 1-3. 処理フロー全体像

```
ユーザーの分析依頼（自然言語）
       ↓
[Planning]  タスクを分解・実行計画を立案
       ↓
[Reasoning] 何を調べるべきか推論（CoT）
       ↓
[Acting]    SQL実行 / Python実行 / API呼び出し
       ↓
[Observing] 結果を観察・評価
       ↓
[Reasoning] 次のアクションを推論（ReActループ）
       ↓ （完了判断まで繰り返し）
[Memory更新] ナレッジベース・履歴の更新
       ↓
分析結果・インサイトをアウトプット
```

### 1-4. 先行事例

#### 🔍 深掘り事例①：メルカリ「Socrates」（ADK）

日本の実例として最も詳細に公開されているのが、メルカリが2025年4月にローンチした社内向けデータ分析AIエージェント **「Socrates（ソクラテス）」** だ。

SQLの知識がない社員でも自然言語で問いかけるだけで分析が完結する。構想からローンチまで約1ヶ月、デモ版は2日で構築、リリース後1週間で約500名が利用し、社内ベンチマークで95%精度を達成している。

実装はGoogle ADK + BigQuery + Vertex AI RAG構成で、成功の鍵はLLMの賢さではなく周辺資産にある。具体的には、JOIN不要なOne Big Table構造でCIで強制されたdescription記述を持つデータマート「Basic Tables」、Socratic Data Catalog、Agent as a Judgeによる継続改善サイクルの3点だ。

```
社員（自然言語）
       ↓
  Socrates（ADK）
       ├─ BigQuery エージェント（SQL生成・実行）
       ├─ Python エージェント（高度な加工・可視化）
       └─ 権限管理・ガードレール
       ↓
  Basic Tables（高品質データマート）
```

参考：[mercan記事](https://careers.mercari.com/mercan/articles/53431/) / [Speaker Deck](https://speakerdeck.com/na0/merukariniokerudetaanariteikusu-ai-eziento-socrates-to-adk-huo-yong-shi-li)

#### 🔍 深掘り事例②：OpenAIの社内データエージェント

OpenAIが2026年1月に公開した「Inside our in-house data agent」は、社員3,500人超・70,000データセット・600PB超の環境で自然言語データ分析を実現したエージェントの報告だ[^5]。

注目すべきは**精度の鍵がモデルではなくコンテキスト設計にある**という点で、6層のコンテキストを管理する設計を採用している。

| Layer | 内容 | 自動/手動 |
|---|---|---|
| 1. テーブル使用状況 | クエリログからの使用頻度・鮮度 | 自動 |
| 2. 人間による注釈 | カラムの意味・使い分けの注意点 | 手動 |
| 3. Codexエンリッチメント | dbt/Airflowのコードから振る舞いを抽出 | 自動 |
| 4. 組織知 | Slack・Notionに散在する暗黙知 | 手動（最も差がつく） |
| 5. メモリ | 対話から蓄積するユーザー別パターン | 半自動 |
| 6. ランタイムコンテキスト | 実行時のテーブル状態・スキーマ変更 | 自動 |

「スキーマ＋RAGだけでは足りない。コンテキストを6層に分けて管理することが精度改善のロードマップになる」というOpenAIの結論は、メルカリSocratesの「Basic Tables → Socratic Data Catalog → データ認証制度」という3段階整備と本質的に同じ話だ。**LLMを作っている会社もデータ基盤のコンテキスト設計で苦労している**という事実は重い。

なお、Workdayとの連携ではコンテキスト設計だけで精度が5倍改善したと報告されている。

参考：山本健太さんによる解説記事（note）https://note.com/pn_yamaken/n/na0e4a3fd44c1

#### その他の主要事例

**Uber QueryGPT**：月14万時間削減・クエリ作成時間10分→3分を達成。ただしground-truthテーブルとの一致率は50%で、「動くSQL」と「正しいSQL」のギャップが依然として課題として残る。

**LinkedIn SQL Bot**（LangGraph）：知識グラフ＋RAG＋マルチエージェント構成。95%が「合格以上」と評価され、「Fix with AI」ボタンが80%のセッションで使用されている。

**Databricks Genie / Snowflake Cortex Analyst**：各DWHにネイティブ統合されたSaaSエージェント。Genie Agent ModeはHP・7-Eleven・SEGA Europeなどが本番採用、Cortex AnalystはBayerなどが採用し90%+の精度を報告している。

**サイバーエージェントDX「AI POS」**、**東京電力EP「V-DAG」**、**NTTドコモ マルチAIエージェントシステム**など、国内でも本番事例が増えている。

### 1-5. 「できること」と「できないこと」

#### できること（2026年現在）

単一テーブル・単純結合ではNL→SQL精度70〜90%に到達している。マルチステップ自律分析（仮説→クエリ→解釈→次の仮説のループ）、Pythonによる統計処理・可視化、レポート・ダッシュボードの自動生成、異常検知・要因分析なども実用レベルに達している。

#### できないこと・限界

**Spider 2.0ベンチマーク**（実エンタープライズSchemaで評価）では、GPT-4oが精度6%まで落ちる。Spider 1.0比で93%もの精度低下が観測されており、「モデルを賢くする」だけでは解決しないことが示されている。

根本的な限界は以下の通りだ。

- **ビジネスルールの推測不可**：`total_amt`が「税抜・割引前」であること、`ord_status=5`が「キャンセル」を意味することは、データだけからは推測できない
- **ハルシネーション**：存在しないテーブル・カラムの参照、誤ったJOINを「もっともらしく」生成する
- **コスト・レイテンシ**：マルチエージェント構成では1問あたり数十回のLLM呼び出しが発生し、レイテンシは10〜30秒に及ぶ
- **暗黙知の補完不可**：Layer 4（組織知）が整備されていないデータ基盤では精度が出ない

**「エージェントは組織のデータの秩序・プロセスの厳密さ・暗黙知の多さを映し出す鏡」**というSocratesチームの言葉が、この限界を端的に表している。

### 1-7. 「精度はモデルではなくコンテキスト設計で決まる」

ここまでの先行事例を横断して見えてくる最重要の知見は一つだ。

**メルカリもOpenAIも同じ結論にたどり着いている**：エージェントの精度はLLMの性能より、そのLLMに渡すコンテキストの設計品質で決まる。

OpenAIの6層設計とSocratesの3段階整備は表現こそ違えど、「スキーマとRAGだけでは足りない。使用状況・ビジネス定義・組織知・コードリネージを段階的に積み上げる必要がある」という同じ主張をしている。

これはWang et al.[^3] が指摘する**Memoryコンポーネントの設計がエージェント品質を左右する**という論文知見と完全に一致する。

---

### 1-8. 2026年のトレンドまとめ

- **「分析」から「行動」へのシフト**：ReAct[^2] の普及により、推奨を出すだけでなく実際にシステムを操作するエージェントが増加
- **マルチエージェント化**：単一エージェントから複数エージェントの協調システムへ（市場シェアは単一58%だがマルチが年率46%で急成長中）
- **コンテキスト設計が競争優位**：LLMの性能競争は収束しつつあり、データ基盤とコンテキスト設計の質が差別化要因になる
- **ガバナンスの整備が本番移行の鍵**：セキュリティ監視・ログ管理・Human-in-the-Loopのフロー整備が、テストから本番移行を分ける

## Part 2: 実際に試してみた

### 2-1. 構成選定の経緯

#### なぜSaaSでは不十分か

Google BigQuery Data Science Agent・Databricks Genie・Snowflake Cortex AnalystのようなマネージドSaaSは導入が早くすぐ使える。しかし業務での本番運用を想定したとき、モデルが固定されカスタマイズが制限されるという本質的な制約がある。

今回の目的は「業務データ基盤と連携した分析エージェントを自前で設計・運用する」ことであり、SaaSではその設計資産が手元に残らない。

#### なぜADKではなくOAI Agents SDKか

先行事例で最も参照実装が充実しているのはADK（メルカリSocrates）だ。しかしADKはGCPインフラと密結合しており、デプロイもVertex AI RAGもBigQuery統合も全てGCP前提になる。

一方、想定する業務環境にはGCP（BigQuery）もAWS（Athena/Redshift）もDatabricks/Snowflakeも混在している。特定クラウドに最適化された設計は、その環境に縛られるトレードオフを生む。

**OAI Agents SDKを選んだ理由は「クラウド・モデル非依存」にある。**

```
OAI Agents SDK（オーケストレーション層）
　× モデル切り替え可能
　　├─ GPT-4o / GPT-5
　　├─ Claude Sonnet 4.6（Anthropic API直接）
　　└─ Gemini 2.5 Flash（互換API経由）
　× データ基盤も切り替え可能
　　├─ BigQuery MCP（GCP環境向け）
　　└─ Athena / Redshift MCP（AWS環境向け）
　× 警察庁交通事故統計オープンデータ（分析題材）
```

この構成なら「同じエージェント設計で、クラウドもモデルも差し替えられる」という点がそのまま設計の価値になる。

#### 先行事例との差別化

今回の取り組みは以下の先行事例と明確に差別化できる。

| | 構成 | 特徴 |
|---|---|---|
| hizaさん | OpenRouter × Gemma 4 | 素のエージェント、コスト最小 |
| soyaokiさん | 素のClaude Code | Skills未設定、課題感で終わり |
| メルカリSocrates | ADK × BigQuery | GCP特化・本番運用実績あり |
| **本記事** | OAI Agents SDK | クラウド・モデル非依存、複数環境で使い回せる設計 |

Socratesは「GCPに最適化することで本番精度を出す」アプローチを選んだ。本記事は「GCPに縛られず設計資産を使い回せる」という別の軸を探る。

#### OAI Agents SDKのUI事情

ADKには`adk web`という組み込みのDev UIがあるが、OAI Agents SDK自体にはDev UIはない。ローカル開発時のトレースはOpenAIダッシュボード（platform.openai.com/traces）で確認できる。本番デプロイ時はStreamlit・FastAPI + htmx・Gradioなどで別途フロントを作る必要がある。

```
開発時：
　OAI Agents SDK
　└─ トレース → OpenAIダッシュボード（クラウド送信）
　　　or LangFuse（OSSで自前ホスト）

デプロイ時：
　OAI Agents SDK（バックエンド）
　× Streamlit / FastAPI + htmx（フロントエンド）
```

ローカルで動かすだけなら`pip install openai-agents`だけで済み、Dockerも不要だ。

### 2-2. データセットについて

分析題材には**警察庁交通事故統計情報オープンデータ**（https://www.npa.go.jp/publications/statistics/koutsuu/opendata/index_opendata.html）を使用した。2024年版で約30万件、本票・補充票・高速票の3ファイル構成でリレーショナルDB的な構造を持つ。

前処理として以下が必要だった。

- **文字コード**：Shift-JIS → UTF-8変換
- **緯度経度**：独自の度分秒形式 → 10進数変換
- **コード体系**：別途配布のコード表とJOINして意味を付与
- **ファイル結合**：本票 × 補充票 × 高速票のマージ

前処理さえ乗り越えると、30万件・数十カラムの分析しがいのある構造化データになる。

### 2-3. コンテキスト設計

OpenAIの6層コンテキスト設計とSocratesの知見を参考に、以下のコンテキストを整備した。

**Layer 1（使用状況）**：BigQueryのINFORMATION_SCHEMA.JOBSからクエリログを収集し、よく参照されるカラムにスコアを付与。

**Layer 2（人間による注釈）**：コード表を変換した「事故類型コード=1は人対車両」のような定義、「致死率を見るには死者数÷事故件数で計算する」といった分析ロジックをYAMLで管理。

**Layer 3（コードリネージ）**：前処理スクリプトから各カラムの変換ロジックを自動抽出。

```yaml
# context/traffic_accident_catalog.yaml（イメージ）
tables:
  - name: honhyo
    description: 交通事故本票。1行が1事故に対応。集計・分析の起点となるテーブル。
    key_columns:
      - name: jiko_ruikei_cd
        description: 事故類型コード。1=人対車両、2=車両相互、3=車両単独
      - name: shisha_su
        description: 死者数。当日死亡のみカウント。30日以内死者は別カラム(sanju_nichi_su)
    notes: |
      honhyo単体でも使えるが、当事者属性を見るにはhojuhyoとJOINが必要
      JOIN条件: honhyo.honshu_no = hojuhyo.honshu_no
```

## おわりに

動向調査から見えてくるのは、データ分析エージェントは「使えるかどうか」ではなく「いかにコンテキストを積み上げ本番運用に持ち込むか」のフェーズに入ったということだ。

メルカリSocratesもOpenAI社内エージェントも、精度の鍵はLLMよりデータ基盤の整備にあると結論づけている。「エージェントは組織のデータの秩序を映し出す鏡」という言葉が、この分野の本質を表している。

Part 2では、OAI Agents SDK × 警察庁オープンデータで実際に試した構成・ハマりどころ・精度感を具体的にレポートする。
